# Implementation Plan: Participant Engagement Features

## Overview
Implement four core features for student participants:
1. **Dynamic Feed** - Filter events by geolocation and interests
2. **Social Proof** - Likes system + enhanced participant visibility
3. **Wallet UI** - Transaction history page
4. **Leaderboard** - Rankings by nation, city, and establishment

## Assumptions & Design Decisions

### Interest System
- **Approach**: Predefined category tags matching the `events.category` field
- **Event matching**: Match event `category` exactly against user `interests` array
- **Rationale**: Ensures clean UX; users select from known categories; easy maintenance
- **Implementation**: Store `interests` as JSON array on `users` table

### Geolocation Filtering
- **Primary filter**: `city` string match (exact or partial) - events filtered by matching city
- **No radius/distance**: Based on user's city field only
- **Rationale**: Simpler implementation; aligns with existing user data; good enough for MVP

### Likes System
- **Model**: `Like` morphable (likeable) to support events, comments, rewards in future
- **Fields**: `user_id`, `likeable_id`, `likeable_type`, timestamps
- **Unique constraint**: A user can only like something once
- **UI**: Heart icon toggle, counter displays

### Social Proof
- **Participants display**: Show first 5 participant avatars in a row; "+N more" button opens modal with full grid view
- **Likes**: Show total like count; "X of your friends liked this" badge if any liked
- **Activity hints**: "Your friend [Name] registered" (optional, could be future enhancement)

### Wallet UI
- Full transaction history table on dedicated `/wallet` page
- Columns: Date, Type, Description, Amount (+/-), Balance After
- Group by month, visualize with simple chart bars
- Filter by type: earned/spent/burned/adjusted

### Leaderboard
- Metrics: `points_balance` (primary), `total_hours` (secondary), `events_participated_count`
- **Three scopes**:
  - **National**: Top 100 users globally
  - **Par Ville**: Top 100 within each city
  - **Par Établissement**: Top 100 per `establishment_id` (school)
- **Time windows**: All-time + weekly (calendar week Monday-Sunday reset)
- Display: Rank, user name (avatar), grade, points, hours, establishment

## Technical Architecture

### Database Changes
1. `likes` table (morphable)
2. Add `interests` JSON column to `users` table

### Controllers
1. **EventController@index** - Enhanced with filters (city, interests, search)
2. **LikeController** - Toggle like/unlike for events
3. **WalletController** - Transaction history page
4. **LeaderboardController** - Rankings with scope/city/establishment filters

### Models
1. **Like** model (morphable)
   - User belongsToMany through Likes for likedEvents
   - Event hasMany Likes
2. Extend **User**:
   - `likedEvents()` relation
   - `interests` JSON cast
3. Extend **Event**:
   - `likesCount()` accessor
   - `isLikedBy(User $user)` method

### Views
1. **events/index.blade.php** - Enhanced with sidebar filters
2. **events/show.blade.php** - Like button, enhanced participants section
3. **wallet/index.blade.php** - New wallet page
4. **leaderboard/index.blade.php** - New rankings page
5. **profile/edit.blade.php** - Add interests field

### Actions
- **LikeEventAction** - Toggle like, encapsulate logic
- **UnlikeEventAction** - Remove like

## Implementation Phase Breakdown

### Phase 1: Database & Foundation (Migrations + Models)
- Create `likes` table migration
- Add `interests` column to `users` via migration
- Create `Like` model
- Update `User` model with interests cast and likes relation
- Update `Event` model with likes relation and accessors
- Run `php artisan pint` for code formatting

### Phase 2: Likes System (Actions + Controllers + Views)
- Create `LikeEventAction` and `UnlikeEventAction`
- Create `LikeController` (toggle method)
- Add routes: `POST /events/{event}/like`, `DELETE /events/{event}/like`
- Add like button to `events/show.blade.php`
- Add like indicator to `events/index.blade.php` cards
- Show liked events filter option

### Phase 3: Dynamic Feed Enhancements
- Enhance `EventController@index` with filters:
  - By city (string match; default to logged-in user's city)
  - By interests matching (events where `category` in user interests array)
  - Search by title/description keyword
- Create filter sidebar component with: city input, interests multi-select, search box
- Update `events/index.blade.php` with filter UI sidebar
- Update user profile to add interests selection (multi-select from predefined categories)
- Events list should default to user's city filter if user has city set

### Phase 4: Social Proof UI
- Enhance participants section in `events/show.blade.php`:
  - Show avatar grid of first 5 participants
  - Show "N others are going" count
  - Show mutual friends badge if applicable
- Add "X friends liked this" section
- Add activity feed: "Your friend [Name] just registered for this event"

### Phase 5: Wallet UI
- Create `WalletController` with `index` method
- Route: `GET /wallet` (middleware auth)
- View: `wallet/index.blade.php` with:
  - Current balance card
  - Transaction history table (pagination)
  - Filter tabs by transaction type
  - Simple bar chart of points earned per month (optional)
- Add wallet link to navigation (for students only)

### Phase 6: Leaderboard System
- Create `LeaderboardController` with methods:
  - `index()` - default (all-time national)
  - `city($city)` - by city
  - `establishment($establishmentId)` - by school
  - `weekly()` - this week's top (calendar week, Monday-Sunday reset)
- Routes:
  - `GET /leaderboard`
  - `GET /leaderboard/city/{city}`
  - `GET /leaderboard/establishment/{establishment}`
  - `GET /leaderboard/weekly`
- Views:
  - `leaderboard/index.blade.php` with tabs for Nation/City/Establishment/Weekly
  - Table with rank, user info, points, hours, grade
  - Badge for top 3 (gold/silver/bronze)
- Add leaderboard link to navigation or dashboard

### Phase 7: Testing
- Write feature tests for:
  - Like/unlike flow
  - Feed filtering by city/interests
  - Wallet transaction display (assert correct data)
  - Leaderboard ranking calculations
  - User can update interests via profile
- Run `php artisan test --compact`

### Phase 8: Polish & Edge Cases
- Add eager loading to prevent N+1 queries
- Add caching for leaderboard (5-minute cache)
- Add clickable badges/cities in leaderboard to drill down
- Add empty states for wallet, leaderboard
- Ensure mobile-responsive filter UI
- Add "My Events" section on student dashboard (upcoming, past)
- Add notifications: friend joined event, like received

## Dependencies & Order

**Must complete before starting a phase:**
- Phase 2 depends on Phase 1 (DB + models)
- Phase 3 depends on Phase 1 (interests column)
- Phase 4 depends on Phase 2 (likes) and Phase 3 (feed)
- Phase 5 independent (uses existing PointsTransaction)
- Phase 6 independent (uses existing user data)
- Phase 7 depends on all implementation completed

# Plan: Event Wishlist Feature

## Objectif
Allow students to save events to a wishlist for future registration.

## Current Flow
- Students can only register for events (`status = 'registered'`)
- No way to "save" or "bookmark" events for later

## Solution
Add a new status `'wishlist'` to the existing `event_user` pivot table and create wishlist routes.

## Implementation

### 1. Migration - Add wishlist status
```php
// In enum status, add 'wishlist' to existing values:
// ['registered', 'checked_in', 'absent', 'cancelled', 'wishlist']
```

### 2. Model - EventUser
Add accessor method:
- `hasSavedToWishlist(): bool` - returns true if status === 'wishlist'

### 3. Routes
Add wishlist endpoints to `routes/web.php`:
```php
Route::post('events/{event}/wishlist', [WishlistController::class, 'store'])->name('events.wishlist');
Route::delete('events/{event}/wishlist', [WishlistController::class, 'destroy'])->name('events.unwishlist');
```

### 4. Controller
Create `WishlistController` or add to `EventRegistrationController`:
- `store`: Save event to wishlist (status = 'wishlist')
- `destroy`: Remove from wishlist
- Authorization: Students only (`event.register` permission)

### 5. Views (Optional)
Add "Save to Wishlist" button on event details page.

## Business Logic
- Wishlist items don't count toward volunteer quota
- User can have multiple wishlist items
- Can convert wishlist to registration later
- Can register directly (skip wishlist) even if previously wished

## Alternative Approach

| Approach | Pros | Cons |
|----------|------|------|
| **Use Status (Recommended)** | Single table, simple query | Need migration |
| New pivot table | More flexible | More complex |

The status approach is recommended as it keeps all event-user relationships in one table.

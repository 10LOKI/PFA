# Plan: Notifications & Messages Privés (Pusher)

## Contexte
L'utilisateur veut ajouter un système de notification en temps réel et de messages privés via Pusher.

## Questions Clarifiantes

1. **Types de notifications attendus ?**
   - Notifications d'événements (nouveau, modifié, annulé)
   - Check-in confirmé
   - Récompenses / Paliers de grade atteints
   - Messages privés entre utilisateurs

2. **Notifications temps réel ou in-app uniquement ?**
   - Temps réel = Pusher (WebSockets)
   - In-app = Laravel Notifications (database)

3. **As-tu déjà un compte Pusher ?** (besoin de credentials APP_ID, APP_KEY, APP_SECRET, CLUSTER)

---

## Plan Technique Proposé

### Phase 1: Configuration Pusher
- Installer `pusher/pusher-php-server`
- Configurer .env avec credentials Pusher
- Créer config/pusher.php

### Phase 2: Notifications (Database + In-App)
- Créer table `notifications` (Laravel default)
- Créer migration pour `conversations` et `messages`
- Créer models: Notification, Conversation, Message
- Créer NotificationChannel pour chaque type

### Phase 3: Messages Privés
- CRUD Conversations (participants)
- CRUD Messages
- API endpoint: /api/messages, /api/conversations

### Phase 4: Frontend (Blade)
- Dropdown notifications dans navigation
- Page messages (/messages)
- Alpine.js pour mise à jour temps réel

### Phase 5: Events Pusher
- Broadcast event créé → notification
- Nouveau message → notification conversation
- Check-in confirmé → notification student

---

## Détails Techniques

### Models à créer
- `App\Models\Conversation` - participants, last_message_at
- `App\Models\Message` - sender_id, conversation_id, body, read_at

### Controllers
- `App\Http\Controllers\NotificationController` - markAsRead, getAll
- `App\Http\Controllers\MessageController` - index, store, show

### Tables SQL
```sql
conversations (id, created_at, updated_at)
messages (id, conversation_id, sender_id, body, read_at, created_at, updated_at)
conversation_user (conversation_id, user_id)
```
